# SEO Roadmap — Geometki

**Goal:** Resolve every technical, on-page, and structural SEO issue blocking or limiting Top 10 rankings on Yandex and Google for Geometki (https://geometki.com) — a Russian-language crowdsourced platform for discovering interesting places across Russia.

**Audit scope:** Next.js 16 codebase at `client/`, live site verification, sitemap and robots.txt inspection, structured data review, i18n analysis. All findings are based on observed code or live-site data — no assumptions.

---

## Critical (fix immediately — blocking indexation or ranking)

- **#1 [sitemap.tsx]**: The sitemap contains only 10 URLs and is missing the homepage (`https://geometki.com/`), the `/tags` page, and all individual place and user pages. The live sitemap confirmed this — dynamic place/user URLs are generated from the API but the `usersPages` builder on line 31 reads from `data?.places` instead of `data?.users`, meaning user profile URLs in the sitemap are duplicates of place URLs. **Fix:** Change line 31 from `data?.places?.map(...)` to `data?.users?.map(...)` to fix the data source bug. Add the homepage (`''`) and `/tags` to the `staticPages` array. Verify the API returns both `places` and `users` arrays and that all place/user IDs are included.

- **#2 [sitemap.tsx]**: The sitemap contains zero hreflang `<xhtml:link rel="alternate">` annotations. Without them, Google cannot determine the relationship between the Russian (`https://geometki.com/places/123`) and English (`https://geometki.com/en/places/123`) versions of the same page, leading to duplicate content penalties and incorrect language targeting. **Fix:** For every URL entry in the sitemap, add two `<xhtml:link>` alternate nodes — one for `ru` pointing to the base URL, one for `en` pointing to the `en/` prefixed URL — plus a `x-default` alternate. Add `xmlns:xhtml="http://www.w3.org/1999/xhtml"` to the `<urlset>` element.

- **#3 [All pages — `_app.tsx`, every page file]**: No `hreflang` link tags are rendered in the HTML `<head>` on any page. The live site fetch of both `https://geometki.com/` and `https://geometki.com/en/` confirmed zero `<link rel="alternate" hreflang="...">` tags. Google and Yandex require these to serve the correct language version to the correct audience and to avoid treating `geometki.com/places` and `geometki.com/en/places` as duplicate pages. **Fix:** Use `next-seo`'s `languageAlternates` prop on every `<NextSeo>` instance. For the Russian default locale, add `[{ hrefLang: 'ru', href: 'https://geometki.com/[path]' }, { hrefLang: 'en', href: 'https://geometki.com/en/[path]' }, { hrefLang: 'x-default', href: 'https://geometki.com/[path]' }]`.

- **#4 [`pages/_app.tsx`]**: Yandex.Metrika and Google Analytics (gtag.js) are injected via `dangerouslySetInnerHTML` inside a plain `<div>`, not using `next/script`. This prevents Next.js from applying the `afterInteractive` or `lazyOnload` loading strategy, causing both analytics scripts to block the main thread and hurt Core Web Vitals (TBT, FID/INP). **Fix:** Replace the `dangerouslySetInnerHTML` block with two `<Script>` components from `next/script`, using `strategy="afterInteractive"` for Yandex.Metrika and `strategy="lazyOnload"` for Google Analytics. Move the Yandex noscript pixel to a separate `<noscript>` element alongside the Script component.

- **#5 [`pages/_app.tsx` — viewport meta]**: The viewport meta tag sets `maximum-scale=1`, which prevents users from zooming in on mobile. Google's mobile-friendliness evaluation explicitly flags `maximum-scale=1` and `user-scalable=no` as accessibility barriers, and accessibility issues directly affect mobile ranking signals. **Fix:** Change the viewport content to `width=device-width, initial-scale=1` — remove `maximum-scale=1` and `shrink-to-fit=no`.

- **#6 [`pages/index.tsx` — canonical URL]**: The canonical URL is built as `SITE_LINK + (i18n.language === 'en' ? 'en' : '')`. For the Russian locale this produces `https://geometki.com/` (correct), but for English it produces `https://geometki.com/en` without a trailing slash, while the actual URL is `https://geometki.com/en/`. This inconsistency between canonical tag and actual URL signals a mismatch to crawlers. The same pattern repeats across index.tsx (line 27 uses `'en'` not `'en/'`). **Fix:** Change `'en'` to `'en/'` in `pages/index.tsx` line 27 to match all other pages.

## High Priority (significant ranking impact)

- **#7 [All pages — `<NextSeo>` title tags]**: No page sets a `titleTemplate` or `defaultTitle` via a `<DefaultSeo>` component in `_app.tsx`. This means each page manually concatenates the site name into the title (e.g. `t('news-feed') + ' - ' + t('interesting-places')`), creating inconsistent formatting. There is also no Twitter Card metadata on any page — confirmed by codebase search showing zero `twitter:` properties. **Fix:** Add a `<DefaultSeo>` component in `_app.tsx` that sets `titleTemplate="%s | Geometки"`, `defaultTitle="Геометки — Интересные места России"`, and Twitter Card defaults (`twitter: { cardType: 'summary_large_image', site: '@geometki' }`). Remove manual site name appending from individual page titles.

- **#8 [All pages]**: No page implements the `<html lang="">` attribute explicitly at the document level. While Next.js i18n injects a `lang` attribute, there is no `pages/_document.tsx` file in the project to set it explicitly or to add any custom `<Head>` structure that persists across navigations. The WebFetch live check did not detect a visible `lang` attribute. **Fix:** Create `pages/_document.tsx` extending `_Document`, which sets `<Html lang={locale}>` using the router locale. This ensures every page has a correct `lang` attribute for both crawlers and accessibility.

- **#9 [All place pages — `pages/places/[id]/index.tsx`]**: The canonical URL for place pages is built as `${canonicalUrl}places/${place?.id}`, using a numeric ID (e.g. `/places/142`). Numeric IDs are not descriptive and provide no keyword signal. Competitor platforms (e.g. Tripadvisor, Tourister) use human-readable slugs. This is a structural limitation that affects both ranking and CTR. **Fix:** Add a `slug` field to the `Place` API model and generate it server-side from `place.title` (transliterated to Latin or kept in Cyrillic, hyphen-separated). Update the router to use `pages/places/[slug].tsx` and set up permanent 301 redirects from old numeric ID URLs to the new slugs. This is a significant effort but provides lasting SEO value.

- **#10 [OG images — `pages/index.tsx`, `pages/map.tsx`, `pages/categories.tsx`]**: Static OG images (`/images/pages/main.jpg`, `/images/pages/map.jpg`, `/images/pages/categories.jpg`) are referenced with relative paths (e.g. `url: '/images/pages/main.jpg'`). The Open Graph specification requires absolute URLs for `og:image`. Social crawlers (Facebook, Telegram, VK) and Yandex may not resolve relative paths, causing link previews to show no image. **Fix:** Prefix all static OG image URLs with the absolute site URL: `url: \`${SITE_LINK}images/pages/main.jpg\``.

- **#11 [`pages/places/[id]/index.tsx` — schema.org]**: Several important fields in the `placeSchema` (LocalBusiness) are commented out: `author`, `dateModified`, and `datePublished`. These are valuable signals for Google's E-E-A-T evaluation and help establish content freshness and authorship. The `interactionStatistic` entry is also missing the required `interactionType` property (`"https://schema.org/ViewAction"`) which makes the schema invalid. **Fix:** Uncomment and restore `author`, `dateModified`, and `datePublished` fields. Add `interactionType: 'https://schema.org/ViewAction'` to the `interactionStatistic` object in both `pages/places/[id]/index.tsx` and `functions/schema.ts`.

- **#12 [`pages/places/[id]/index.tsx` — schema.org type]**: Places are typed as `LocalBusiness` in structured data, but many POIs (waterfalls, mountains, abandoned buildings, forests, lakes) are not businesses. Using the wrong schema type confuses Google's entity understanding and blocks eligibility for Knowledge Panel and rich result features specific to geographical POIs. **Fix:** Use `TouristAttraction` as the primary `@type` for natural and historical POIs, or use a typed union `['TouristAttraction', 'Place']`. Add category-based type mapping: `nature` → `NaturalHeritage`, `architecture` → `LandmarksOrHistoricalBuildings`, etc. Reference the place's category to select the most appropriate schema type.

- **#13 [`pages/places.tsx` — OG tags]**: The `/places` listing page has no `siteName`, `title`, or `url` in its `openGraph` object — only `images`, `locale`, and `type`. This means OG shares of the places listing will show no title or site name, hurting social traffic and brand recognition in link previews. **Fix:** Add `siteName: t('geotags')`, `title: title`, `url: canonicalPage`, and `description` to the `openGraph` object in `pages/places.tsx`.

- **#14 [`pages/map.tsx` — indexability]**: The map page loads all interactive map components client-side via `dynamic(..., { ssr: false })`. While the `<NextSeo>` metadata and page title are server-rendered, crawlers visiting `/map` receive almost no meaningful text content — no place names, no descriptions, no H2 headings beyond the title. The page has significant ranking potential for queries like "карта интересных мест России". **Fix:** Add a server-rendered section below the map (visible or SEO-only within normal page flow) that lists the most recently added or most popular places with their names, categories, and locations. This can coexist with the interactive map and provide crawlable content.

- **#15 [All pages — title character length]**: Several page titles are either too short or potentially too long when combined with the site name. For example, `t('features-of-places')` translates to "Особенности мест" (16 characters) — too short to be competitive. The categories page title "Категории мест" (14 characters) is similarly thin. **Fix:** Expand page titles to 50–60 characters including the site name: "Категории геометок — интересные места России | Геометки" for the categories page; "Особенности и теги геометок — Геометки" for the tags page.

## Medium Priority (incremental improvements)

- **#16 [`pages/tags.tsx` — missing OG tags]**: The tags page has only `title`, `canonical`, and `description` in `<NextSeo>` — no `openGraph` object at all. This means VK, Telegram, and other social shares of `/tags` will show no image or structured preview. **Fix:** Add an `openGraph` block with `title`, `description`, `type: 'website'`, `url`, `locale`, `siteName`, and the categories OG image as a fallback.

- **#17 [`pages/users.tsx`, `pages/users/[id]/*.tsx` — missing OG tags]**: The users listing page has no `openGraph` object in `<NextSeo>`. User sub-pages (`/users/[id]/places`, `/users/[id]/photos`, `/users/[id]/bookmarks`) also lack `openGraph` tags entirely. **Fix:** Add `openGraph` blocks to all user pages, using the user's avatar as the OG image where available.

- **#18 [`pages/sitemap.tsx`]**: The sitemap uses `changefreq: 'monthly'` for all static pages and `changefreq: 'daily'` for all place pages. The homepage and news-feed page likely update multiple times per day. Conversely, some older places may not update frequently. Using inaccurate `changefreq` values can cause inefficient crawl budget allocation. **Fix:** Set `changefreq: 'daily'` and `priority: '1.0'` for the homepage. Use `changefreq: 'weekly'` for category and tag pages. For place pages, derive `changefreq` from the `updated` date — if updated within 7 days use `'daily'`, within 30 days use `'weekly'`, otherwise `'monthly'`.

- **#19 [`robots.txt`]**: The robots.txt has no `Disallow` rules, allowing crawlers to waste budget on pages that should not be indexed (e.g. `/auth`, `/places/create`, `/users/settings`, `/unsubscribe`, `/places/*/edit`). These pages already have `noindex` in code but blocking them in robots.txt prevents crawl budget waste entirely. **Fix:** Add disallow rules: `Disallow: /auth`, `Disallow: /places/create`, `Disallow: /*/edit`, `Disallow: /users/settings`, `Disallow: /unsubscribe`. Add a Yandex-specific `Crawl-delay: 1` entry under a `User-agent: YandexBot` block to prevent overloading the server.

- **#20 [`pages/places.tsx` — pagination canonicals]**: Paginated pages (`/places?page=2`) correctly include the page number in the canonical URL. However, there is no `rel="prev"` and `rel="next"` link tag implementation, which helps Yandex (which still uses these) understand the pagination sequence. **Fix:** Add `<link rel="prev">` and `<link rel="next">` inside a `<Head>` block on paginated pages for `/places`, `/users`, and user sub-pages that paginate.

- **#21 [`components/common/places-list/PlacesListItem.tsx`]**: Category icon images use `alt={''}` (empty alt). While decorative images can have empty alt text, the category icon is a meaningful visual indicator (e.g. showing "waterfall" or "mountain"). Empty alt text removes a keyword signal from the image. **Fix:** Set the alt text to the place category title: `alt={place.category?.title || ''}`.

- **#22 [`components/pages/place/place-header/PlaceHeader.tsx`]**: The place cover image is rendered as a CSS `background-image` in two `<div>` elements (desktop and mobile), not as an `<img>` or Next.js `<Image>` component. Background images are not indexed by Google Image Search and provide no `alt` text signal for the place's primary visual. This is a missed opportunity for image traffic and place discovery. **Fix:** Replace the `background-image` CSS approach with a `<Image>` component from `next/image`, setting `alt={place?.title || ''}` and `priority={true}` for the above-the-fold LCP element. Apply `object-fit: cover` via CSS to maintain the same visual appearance.

- **#23 [All pages — LCP image optimization]**: No `priority` prop is set on any hero or above-the-fold `<Image>` component across the codebase. The `PlacesListItem` card images and the place header cover are the likely LCP elements on most pages, but without `priority={true}`, Next.js lazy-loads them, significantly hurting LCP Core Web Vital scores. **Fix:** Add `priority={true}` to the first image rendered on: place list cards for the first 3 items on `pages/places.tsx`, the cover image on `pages/places/[id]`, and the carousel first image on `pages/index.tsx`.

- **#24 [`pages/places/[id]/index.tsx` — BreadcrumbList schema]**: The last item of the `BreadcrumbList` schema has no `item` property (only `name` and `position`). Google's structured data documentation requires the `item` property (a URL) on all list items except the last one on the current page — but the absence is only optional for the last item if the page is the destination. However, the schema also uses `${canonicalUrl}places` for the parent item, which will be `https://geometki.com/places` — correct — but should also include the canonical URL of the place page itself for completeness. **Fix:** Verify the BreadcrumbList renders correctly in Google's Rich Results Test. Add `item: pagePlaceUrl` to the second breadcrumb item if rich result testing shows an error.

- **#25 [`pages/places.tsx` — filter URLs and crawlability]**: Filter URLs like `/places?category=nature&region=123` are server-rendered and indexable, which is correct. However, `/places?lat=55.7&lon=37.6&sort=distance&order=ASC` (coordinate-based sort) URLs are also canonicalized and potentially indexable, creating thin near-duplicate content. The canonical for places includes `lat` and `lon` being excluded but the URLs with those params could still get indexed via internal links. **Fix:** Add `lat`, `lon`, `sort`, and `order` to the `canonicalPage` exclusion (they are already excluded in the code — verify this is applied in the rendered `<link rel="canonical">` tag by testing a live filtered URL). Confirm `noindex` is not needed for pure sort/order parameter combinations.

- **#26 [`pages/users/[id]/bookmarks.tsx`]**: The user bookmarks page is publicly accessible and server-rendered. It will accumulate indexable URLs for every user. Bookmarks are personal and do not represent unique discovery content — they risk creating thousands of thin, near-duplicate pages. **Fix:** Add `noindex={true}` to the `<NextSeo>` in `pages/users/[id]/bookmarks.tsx` and add `Disallow: /users/*/bookmarks` to `robots.txt`.

- **#27 [`components/common/app-layout/footer/Footer.tsx`]**: The footer contains an external link to `https://miksoft.pro` with no `rel="nofollow"` or `rel="noopener"` attribute. While this is a low-stakes issue, passing PageRank to an unrelated personal site is not ideal. **Fix:** Add `rel="nofollow noopener noreferrer"` to the external footer link.

- **#28 [`components/pages/place/place-description/PlaceDescription.tsx`]**: Tag links render as `<Link href="/places?tag=tag_name" title="">` with an empty `title` attribute. The `title` attribute is a minor keyword signal but consistently empty titles represent a missed opportunity. **Fix:** Set `title={`${t('interesting-places')} #${tag}`}` on each tag link.

## Yandex-Specific

- **#29 [Yandex Webmaster]**: There is no Yandex Webmaster site verification meta tag or HTML file in the codebase or `public/` directory. Without Yandex Webmaster verification, it is impossible to monitor Yandex indexation, submit sitemaps to Yandex directly, diagnose Yandex-specific crawl errors, or use Yandex Turbo Pages. **Fix:** Add the Yandex Webmaster ownership verification `<meta name="yandex-verification" content="[code]">` tag in `pages/_app.tsx` inside the `<Head>` block, or place the HTML verification file in `public/`. Register the site at https://webmaster.yandex.ru and submit the sitemap.

- **#30 [Yandex Webmaster — regional targeting]**: Geometki targets places across Russia but has no `<meta name="geo.region" content="RU">` or Yandex Webmaster regional binding configured. Yandex heavily weights regional relevance for geographic queries. **Fix:** Add `<meta name="geo.region" content="RU">` and `<meta name="geo.country" content="Russia">` in `pages/_app.tsx`. For individual place pages, add `<meta name="geo.placename" content={place.address?.region?.name || place.address?.locality?.name}>` and `<meta name="geo.position" content="{lat};{lon}">`.

- **#31 [Yandex.Metrika — injection method]**: Yandex.Metrika is injected via `dangerouslySetInnerHTML` (see issue #4). Beyond the Core Web Vitals concern, this method can occasionally fail on React hydration, causing the counter to not initialize. This would result in missing behavioral data that Yandex uses as a ranking signal (bounce rate, time on site). **Fix:** (Same as #4) Move to `next/script strategy="afterInteractive"`. Ensure the `webvisor: true` setting is preserved as Yandex WebVisor session data is a ranking input for Yandex's behavioral algorithm.

- **#32 [Yandex Turbo Pages]**: Yandex Turbo Pages can significantly accelerate mobile ranking in Yandex for content pages. Individual place pages (`/places/[id]`) are the best candidates — they have rich content, photos, and structured information. There is no Turbo Pages implementation. **Fix:** Implement a Yandex Turbo RSS feed at `/turbo.xml` (or use the Yandex Turbo API) that outputs place pages in Turbo format. Each entry should include the place title, description, photo, and address. Submit the feed URL in Yandex Webmaster under "Turbo Pages".

- **#33 [Yandex — OpenSearch / site search]**: There is no `<link rel="search" type="application/opensearchdescription+xml">` tag, which Yandex uses to enable sitelink search boxes in search results. The site has a functional search (`global-search-placeholder` in translations). **Fix:** Create `public/opensearch.xml` describing the search endpoint and add `<link rel="search" type="application/opensearchdescription+xml" title="Геометки" href="/opensearch.xml">` to `pages/_app.tsx`.

## Structured Data

- **#34 [`functions/schema.ts` + `pages/places/[id]/index.tsx`]**: The `PlaceSchema` function used on the homepage and places listing serializes `place.content` (raw Markdown) directly into the `description` field without stripping Markdown syntax. The detail page correctly uses `removeMarkdown(place?.content)`, but `functions/schema.ts` line 27 uses `place.content` unprocessed. Markdown characters (`**`, `##`, `[]()`) in structured data descriptions are invalid and will trigger Google Search Console schema errors. **Fix:** Apply `removeMarkdown(place.content)` in the `PlaceSchema` function in `functions/schema.ts`.

- **#35 [All place pages — missing `@id` in schema]**: Neither the `PlaceSchema` in `functions/schema.ts` nor the inline `placeSchema` in `pages/places/[id]/index.tsx` includes an `@id` property. The `@id` (a canonical URL) is required for Google to correctly resolve the entity and connect structured data across pages. Without it, Google cannot build a consistent entity graph for places. **Fix:** Add `'@id': \`${SITE_LINK}places/${place.id}\`` to both `PlaceSchema` and the inline `placeSchema` object.

- **#36 [All place pages — missing `url` in schema]**: The `LocalBusiness` / `TouristAttraction` schema on place pages has no `url` property pointing to the canonical page URL. This is a required field for business/place rich results. **Fix:** Add `url: \`${SITE_LINK}places/${place.id}\`` to both schema objects.

- **#37 [`pages/index.tsx` + `pages/places.tsx` — array schema format]**: On the homepage, the LD+JSON script contains a JSON array (`[...placesList.map(...), ...usersList.map(...)]`) rather than individual `<script>` blocks or a `@graph` construct. Embedding an array at the top level of a JSON-LD script is valid per the spec, but Google's documentation recommends either a single `@graph` object or separate `<script>` tags per entity. Mixing `LocalBusiness` and `Person` entities in one array is valid but may confuse entity association. **Fix:** Convert to a `@graph` pattern: `{ "@context": "https://schema.org", "@graph": [ ...entities ] }` or output separate `<script type="application/ld+json">` tags per entity type.

- **#38 [Missing `WebSite` schema]**: No `WebSite` schema with `SearchAction` is present on any page. This is required to be eligible for Google Sitelinks Searchbox and is also read by Yandex for site-level entity understanding. **Fix:** Add a `WebSite` schema block in `pages/_app.tsx` or the homepage: `{ "@type": "WebSite", "@id": "https://geometki.com/#website", "url": "https://geometki.com/", "name": "Геометки", "description": "...", "inLanguage": ["ru", "en"], "potentialAction": { "@type": "SearchAction", "target": { "@type": "EntryPoint", "urlTemplate": "https://geometki.com/places?search={search_term_string}" }, "query-input": "required name=search_term_string" } }`.

- **#39 [`pages/users/[id]/index.tsx` — ProfilePage schema]**: The `ProfilePage` schema is correctly implemented but the `mainEntity` Person object is missing a `url` property. Without a `url`, Google cannot dereference the entity to a canonical person page. **Fix:** Add `url: \`${canonicalUrl}users/${user?.id}\`` to the `mainEntity` object inside `userSchema`.

- **#40 [Category pages — missing ItemList schema]**: The `/categories` page lists all place categories but has no `ItemList` schema marking up the category list. This is a missed opportunity for breadcrumb-style rich results and category discovery in search. **Fix:** Add an `ItemList` schema to `pages/categories.tsx` where each `ListItem` links to `/places?category={category.name}` with the category title as the name.

## Content and i18n

- **#41 [i18n — missing translations]**: The EN locale (`locales/en/common.json`) is missing the key `notification-list-has-been-cleared` (present in RU) and has two extra keys (`input_description`, `notification-list-cleared`) not present in RU. The RU locale has 230 keys vs 231 in EN. While this does not directly affect SEO, translation gaps can cause UI strings to fall back to key names in English, hurting E-E-A-T signals for Russian users. **Fix:** Add `"notification-list-has-been-cleared": "Список уведомлений очищен"` as an alias in EN, or rename the EN key to match RU. Sync `input_description` into RU.

- **#42 [`pages/tags.tsx` — no OpenGraph tags]**: The `/tags` page has `<NextSeo>` with only `title`, `canonical`, and `description` — no `openGraph` block. This means the page produces no `og:title`, `og:description`, or `og:image` tags when shared on VK, Telegram, or other social platforms. This also means the tags page is excluded from any social-driven discovery. **Fix:** Add a full `openGraph` object matching the pattern used in `pages/categories.tsx`, including a fallback OG image from `/images/pages/categories.jpg`.

- **#43 [English content quality — `pages/users/levels.tsx`]**: The EN translation `user-levels-description-1` contains machine-translated awkward phrasing: "the ability to edit geotags other users" (missing "of") and "List of perks available at each level , is constantly expanding" (errant comma and space). Poor English content hurts E-E-A-T for English-language Google ranking. **Fix:** Proofread and rewrite the EN descriptions in `locales/en/common.json` for the `user-levels-description-1` and `user-levels-description-2` keys.

- **#44 [Place page — H1 context and keyword depth]**: Place pages use `place?.title` as both the `<NextSeo title>` and the `<h1>` in `PlaceHeader`. The title is typically a short proper noun (e.g. "Сусанинский водопад"). There is no programmatic enrichment of the title tag with location or category context (e.g. "Сусанинский водопад — водопад в Башкирии"). A bare place name with no surrounding keyword context competes poorly against established travel content. **Fix:** Enrich the title tag in `pages/places/[id]/index.tsx` to: `${place?.title} — ${place?.category?.title} в ${place?.address?.region?.name || place?.address?.locality?.name}`. Keep the H1 as the bare place title for UX, but use the enriched version only in `<NextSeo title>`.

- **#45 [Place description meta — truncation]**: The `description` meta on place pages is generated with `truncateText(removeMarkdown(place?.content), 300)`. A 300-character description is too long for a meta description (Google truncates at ~155 characters, Yandex at ~200). The excess characters are invisible but may cause Google to auto-generate its own snippet from page content instead of using the supplied description. **Fix:** Change the truncation limit from `300` to `155` characters in `pages/places/[id]/index.tsx` to ensure the meta description is fully displayed in both Google and Yandex SERPs.

- **#46 [`pages/places.tsx` — meta description format]**: The places listing meta description is generated as: `${title} - ${placesList.map(({ title }) => title).join(', ').substring(0, 220)}`. This produces a machine-like comma-separated list of place names with no natural language context (e.g. "Интересные места - Заброшенная ВЧ, Музей гражданской авиации, ..."). It is not user-readable and will not attract clicks from SERPs. **Fix:** Write a template-based description: `"${title}: ${placesCount} геометок по всей России. Находите интересные места: заброшки, природу, памятники, водопады. Фотографии, координаты, отзывы."` Use the dynamic `placesCount` value and category/filter context if active.
