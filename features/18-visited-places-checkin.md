# Feature 18 — Verified Visit Mechanic ("Я здесь был")

## Overview

Доработка существующей функции отметки посещения места. Текущая реализация — простой toggle без каких-либо данных о времени, координатах или верификации. Цель: добавить временну́ю метку, опциональную GPS-верификацию и конфигурацию радиуса на уровне места, не ломая существующее поведение и не блокируя пользователей без геолокации.

**Effort:** Low–Medium (2–3 дня backend + 2 дня frontend)
**Impact:** Medium — закрывает технический долг, усиливает ачивки, даёт фундамент для Feature 17 (Fog of War) и Feature 11 (Ghost Places)

---

## Текущее состояние

```php
// Visited.php::set() — весь механизм сейчас:
$insertData = ['user_id' => $this->session->user?->id, 'place_id' => $input->place];
$visitedModel->where($insertData)->first()
    ? $visitedModel->delete(...)   // уже был — убираем
    : $visitedModel->insert(...);  // не был — ставим
```

Таблица `users_visited_places` содержит только `id`, `user_id`, `place_id`. Нет:
- временно́й метки визита (`visited_at`)
- координат в момент клика
- флага верификации
- возможности отличить "реальный" визит от случайного клика

---

## Проблемы, которые решает фича

### 1. Недостижимые места
Точка стоит в центре водохранилища, на острове или в кратере — физически невозможно попасть в радиус 50 м. Решение: флаг `verification_exempt` на месте + политика размещения точки у точки доступа (берег, смотровая площадка), а не в геометрическом центре объекта.

### 2. Нет интернета в месте посещения
Ущелье, подземный объект, удалённая локация — GPS работает без интернета, но отправить запрос нельзя. Решение: офлайн-очередь в `localStorage` (`{place_id, lat, lon, timestamp}`) с отправкой при восстановлении связи. Если прошло более 24 часов с момента фиксации — верификация не принимается (защита от злоупотреблений).

### 3. Геолокация не разрешена
Самый частый кейс на вебе. Решение: мягкий UX-запрос без давления, обычный визит полноценно засчитывается во все метрики.

---

## Философия дизайна: доверие + опциональная верификация

Принудительная GPS-верификация убьёт конверсию на вебе. Большинство игроков на рынке (Foursquare в начале, OpenStreetMap) работают по доверию. Жёсткая верификация оправдана только там, где есть реальное соревнование или злоупотребления.

**Двухуровневая модель:**

| Тип визита | Как получить | В метрике `places_visited` | Отображение |
|---|---|---|---|
| Обычный | Клик без GPS | ✅ засчитывается | стандартная иконка |
| Верифицированный | Клик с GPS в радиусе места | ✅ засчитывается | метка ✓ |

Обе разновидности полностью засчитываются в ачивки (`places_visited`). Верификация — это дополнительный сигнал доверия, а не пропуск.

## Изменения в базе данных

### `users_visited_places`

```sql
ALTER TABLE users_visited_places
    ADD COLUMN visited_at  DATETIME     NOT NULL DEFAULT NOW()   AFTER place_id,
    ADD COLUMN verified    TINYINT(1)   NOT NULL DEFAULT 0       AFTER visited_at,
    ADD COLUMN lat         DECIMAL(10,7) NULL                    AFTER verified,
    ADD COLUMN lon         DECIMAL(10,7) NULL                    AFTER lat;
```

- `visited_at` — нужен для seasonal-ачивок (считать визиты только в окне сезона), для Fog of War (Feature 17) и аналитики
- `verified` — флаг GPS-верификации; `0` по умолчанию, не ломает существующие записи
- `lat / lon` — координаты в момент клика; `NULL` если геолокация недоступна или не запрошена

### `places`

```sql
ALTER TABLE places
    ADD COLUMN visit_radius_m       SMALLINT UNSIGNED NOT NULL DEFAULT 200  AFTER updated_at,
    ADD COLUMN verification_exempt  TINYINT(1)        NOT NULL DEFAULT 0    AFTER visit_radius_m;
```

- `visit_radius_m` — настраиваемый радиус верификации в метрах. Default 200 м. Для крупных объектов (парк, лес, пляж) — 500–2000 м. Для конкретной точки (кафе, памятник) — 50–100 м.
- `verification_exempt` — флаг для мест, которые физически недостижимы или находятся там, где GPS нестабилен. Для таких мест верифицированный визит невозможен в принципе; обычный засчитывается без вопросов.

## Изменения API

### `PUT /visited`

**Запрос — добавляется опциональное поле:**

```json
{
    "place": "abc123",
    "lat": 55.7512,
    "lon": 37.6184
}
```

**Серверная логика верификации:**

```php
$verified = false;

if (!empty($input->lat) && !empty($input->lon) && !$placesData->verification_exempt) {
    $distance = $this->haversineDistance(
        $input->lat, $input->lon,
        $placesData->lat, $placesData->lon
    );
    $verified = $distance <= $placesData->visit_radius_m;
}

$visitedModel->insert([
    'user_id'    => $userId,
    'place_id'   => $input->place,
    'visited_at' => date('Y-m-d H:i:s'),
    'verified'   => $verified ? 1 : 0,
    'lat'        => $input->lat ?? null,
    'lon'        => $input->lon ?? null,
]);
```

**Ответ:**

```json
{
    "visited": true,
    "verified": true
}
```

## Frontend UX

### Кнопка на странице места

**Сценарий 1 — геолокация включена:**
При клике на "Я здесь был" — сразу делается запрос с координатами. Всплывает тост с результатом:
- Верификация прошла: _"Визит подтверждён ✓ +5 XP"_
- Не в радиусе: _"Отмечено без подтверждения — вы не в радиусе места"_

**Сценарий 2 — геолокация не включена / не запрошена:**
При клике — появляется небольшая модалка:

```
Поделитесь геолокацией, чтобы подтвердить визит и получить +5 XP.
Или отметьте без подтверждения.

[Поделиться местоположением]   [Отметить без подтверждения]
```

Пользователь может закрыть — визит засчитывается как обычный.

**Сценарий 3 — `verification_exempt = 1`:**
Кнопка работает без попытки запросить геолокацию. Нет смысла верифицировать место, куда физически невозможно попасть.

### Счётчик визитов на карточке/странице места

Было: `34 посещения`
Стало: `34 посещения · 12 подтверждённых ✓`

### Офлайн-очередь (важно для мобилки)

```typescript
// При клике — фиксируем локально
const pendingVisit = { placeId, lat, lon, timestamp: Date.now() };
localStorage.setItem('pending_visits', JSON.stringify([...existing, pendingVisit]));

// При восстановлении связи — отправляем
window.addEventListener('online', flushPendingVisits);

// Отказываем в верификации, если прошло > 24 часов
const MAX_OFFLINE_AGE_MS = 24 * 60 * 60 * 1000;
```

## Интеграция с ачивками

Метрика `places_visited` в `AchievementsLibrary` — считает **все** визиты (verified + unverified). Это не меняется, чтобы не обесценить уже набранный прогресс.

В будущем (Feature 11 — Ghost Places, где верификация критична) появится отдельная метрика `places_verified` для специализированных ачивок.

| Ачивка | Метрика | Изменение |
|---|---|---|
| `wanderer` Bronze/Silver/Gold | `places_visited` | без изменений |
| `trailblazer` Bronze/Silver/Gold | `places_visited` | без изменений |
| `scout` (none) | `places_visited >= 1` | без изменений |
| _Будущее: `field_agent`_ | `places_verified >= 10/50/200` | **новая метрика** |

---

## Интеграция с Feature 17 (Fog of War)

Feature 17 использует `sessions_history` для пассивного отслеживания движения. `visited_at` из этой фичи служит дополнительным источником для подтверждения "First Explorer" статуса: если пользователь нажал "Я здесь был" с координатами раньше, чем его `sessions_history` попал в тайловый кронджоб — визит засчитывается в Fog of War.

## Deployment

1. Миграция: добавить колонки в `users_visited_places` и `places` (не ломающие изменения — все новые поля имеют DEFAULT)
2. Обновить `Visited.php::set()` — принять опциональные `lat/lon`, добавить логику верификации
3. Обновить `Visited.php::place()` — вернуть `verified_count`
4. Обновить `UsersVisitedPlacesModel` — добавить новые поля в `$allowedFields`
5. Frontend: добавить модалку с запросом геолокации, обновить тост, счётчик визитов

Существующие записи в `users_visited_places` сохраняются с `verified = 0`, `visited_at = created_at` (если нет — текущая дата при миграции).
